# 모델 가정 문서

> 각 구성요소가 모델의 어떤 한계를 보완하는지 명시한다.
> 새 모델 출시 시 이 문서를 기준으로 하네스 구성요소의 필요성을 재평가한다.

## 기준 모델

- **현재 기준**: Claude Opus 4.6 (2026-03 기준)
- **마지막 재평가**: 2026-03-27

## 구성요소별 모델 가정

| 구성요소 | 보완하는 모델 한계 | 재평가 시점 |
|---------|-----------------|-----------|
| PM (이슈 분해) | 대규모 스펙을 실행 가능한 단위로 분해하는 능력. 의존성 관계와 병렬 처리 가능성을 정확히 판단하는 능력. | 모델이 복잡한 프로젝트 분해를 자율적으로 수행할 때 |
| Architect (설계 + 테스트 시나리오) | 인터페이스 정의와 테스트 시나리오를 사전에 작성하는 능력. 구현 전 설계 결정을 체계적으로 문서화하는 능력. | 모델이 TDD를 자율적으로 수행하고, 설계 없이 일관된 구현을 보장할 때 |
| FE/BE 분리 개발 | 병렬 개발 시 API 계약 기반으로 독립적으로 작업하는 능력. 단일 에이전트가 풀스택을 일관되게 처리하는 한계. | 모델이 대규모 풀스택을 일관되게 처리할 때 |
| Evaluator (Auditor + Reviewer 통합) | 자기 평가 편향 (self-evaluation bias). 자신의 작업을 평가하면 객관성을 잃는 현상. | 자기 평가 정확도가 외부 평가와 동등해질 때 |
| QA (E2E 듀얼 뷰포트) | 브라우저 테스트 자동화의 품질. 모바일/데스크톱 뷰포트별 반응형 검증 능력. | 모델의 E2E 테스트 신뢰도가 높아질 때 |
| 적응형 복잡도 | 이슈 규모에 따른 최적 프로세스 깊이 판단 능력. | 모델이 스스로 프로세스 깊이를 결정할 수 있을 때 |
| 부정 편향 지시 | 기본적으로 관대하게 평가하는 경향. 문제를 발견해도 "괜찮다"고 자기설득하는 현상. | 모델이 기본적으로 엄격한 평가 태도를 보일 때 |

## 재평가 프로토콜

1. 새 모델 출시 시 이 문서를 검토한다
2. 각 구성요소에 대해 **제거 실험**을 수행한다:
   - 구성요소를 제거한 상태에서 동일 태스크를 실행
   - 결과 품질을 비교
   - 품질 저하가 없으면 해당 구성요소 제거 검토
3. 한 번에 하나의 구성요소만 제거하여 개별 영향을 측정한다
