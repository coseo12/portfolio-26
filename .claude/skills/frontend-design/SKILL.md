# 프론트엔드 디자인

제너릭한 AI 미학을 피하고, 독창적이며 프로덕션 수준의 프론트엔드 인터페이스를 구현한다.

> 참고: [Anthropic frontend-design 플러그인](https://github.com/anthropics/claude-code/tree/main/plugins/frontend-design) 기반

## 디자인 사고 (코딩 전 필수)

코드를 작성하기 전에 다음 4가지를 먼저 정의한다:

### 1. Purpose (목적)
- 이 인터페이스의 목적은 무엇인가?
- 주요 사용자는 누구인가?

### 2. Tone (미학적 방향)
아래에서 명확한 방향을 선택한다. 중간을 택하지 않는다:

| 방향 | 특징 |
|------|------|
| 극단적 미니멀리즘 | 최소 요소, 여백 극대화, 타이포그래피 중심 |
| 맥시멀리즘 | 풍부한 시각 요소, 레이어, 텍스처 |
| 레트로-미래주의 | 복고풍 + 미래적 요소 혼합 |
| 유기적/자연 | 부드러운 곡선, 자연 모티프, 흐르는 레이아웃 |
| 럭셔리/정제 | 세련된 타이포, 절제된 색상, 고급스러운 인상 |
| 장난스러움 | 밝은 색상, 재미있는 인터랙션, 친근한 느낌 |
| 에디토리얼/매거진 | 강한 타이포 위계, 그리드, 사진 중심 |
| 브루탈리즘 | 거친 타이포, 날것의 레이아웃, 의도된 불편함 |
| 아르데코/기하학적 | 기하학 패턴, 금속 색상, 대칭 구조 |
| 소프트/파스텔 | 파스텔 톤, 둥근 모서리, 부드러운 그라데이션 |
| 산업/실용적 | 기능 우선, 모노스페이스 폰트, 정보 밀도 높음 |

### 3. Constraints (제약)
- 사용 프레임워크 (React, Vue, Svelte 등)
- 성능 요구사항
- 접근성 수준 (WCAG AA/AAA)
- 반응형 범위

### 4. Differentiation (차별점)
- 이 인터페이스가 기억될 한 가지 요소를 정의한다

## 구현 가이드라인

### 타이포그래피
- **독특하고 캐릭터 있는 폰트**를 선택한다
- 피해야 할 폰트: Arial, Inter, Roboto, Open Sans 등 과도하게 사용된 폰트
- **전략**: 인상적인 디스플레이 폰트 + 정제된 본문 폰트 조합
- 폰트 크기, 굵기, 자간을 의도적으로 설정한다

### 색상과 테마
- 명확한 미학에 완전히 커밋한다 — 안전한 중간을 택하지 않는다
- CSS 변수로 일관성을 유지한다
- **전략**: 강한 주조색 + 날카로운 포인트 색상
- 소극적인 팔레트를 피한다

### 모션과 애니메이션
- HTML 프로젝트: CSS-only 솔루션 우선
- React 프로젝트: Motion 라이브러리 활용 가능
- **고영향 전략**:
  - 페이지 로드 시 한 번의 조직된 애니메이션
  - 스크롤 트리거 인터랙션
  - 예상치 못한 호버 상태
- 과도한 애니메이션보다 적절한 타이밍과 이징이 중요하다

### 공간 구성
- 예상치 못한 레이아웃과 비대칭을 활용한다
- 오버랩, 대각선 흐름, 그리드 깨기 요소
- 여유로운 네거티브 스페이스 또는 통제된 밀도
- 모든 요소가 동일한 그리드에 갇힐 필요 없다

### 배경과 시각 디테일
- 단색 배경 대신 분위기와 깊이를 생성한다
- 활용 기법:
  - 그라데이션 메시, 노이즈 텍스처, 기하학 패턴
  - 레이어드 투명도, 드라마틱한 그림자
  - 데코레이티브 보더, 커스텀 커서, 그레인 오버레이

## 피해야 할 것 (제너릭 AI 미학)

다음은 AI 생성 UI의 대표적인 함정이다. 의식적으로 피한다:

- ❌ 과도하게 사용된 폰트 (Inter, Roboto, Arial)
- ❌ 뻔한 색상 조합 (흰 배경에 보라색 그라데이션)
- ❌ 예측 가능한 레이아웃 (히어로 → 3열 카드 → CTA → 푸터)
- ❌ 의미 없는 그라데이션 배경
- ❌ 기본 border-radius와 box-shadow
- ❌ 문맥을 무시한 일반적 선택

## 접근성 필수 사항

디자인이 독창적이더라도 접근성은 반드시 준수한다:

- 충분한 색상 대비 (WCAG AA 이상)
- 키보드 내비게이션 완전 지원
- 시맨틱 HTML 사용
- ARIA 속성 적절히 적용
- 모션: `prefers-reduced-motion` 미디어 쿼리 대응
- 스크린 리더 호환

## 워크플로우

1. **디자인 사고** — Purpose, Tone, Constraints, Differentiation 정의
2. **Architect 설계 확인** — 컴포넌트 구조, 디자인 시스템 규칙 참조
3. **Planner 기획 확인** — 화면 흐름, 핵심 인터랙션 참조
4. **구현** — 위 가이드라인에 따라 구현
5. **검증** — 접근성, 반응형, 브라우저 호환성 확인

## 규칙

- 코딩 전 디자인 사고 4단계를 반드시 거친다
- 미학적 방향을 선택했으면 끝까지 일관되게 밀고 나간다
- "안전한 기본값"보다 "의도된 선택"을 우선한다
- 독창성과 접근성은 양립할 수 있다 — 둘 다 포기하지 않는다
- Architect의 디자인 시스템 규칙이 있으면 그 범위 안에서 독창성을 발휘한다
