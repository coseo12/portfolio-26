# 에러 수정

CI/QA 실패 로그를 파싱하고, 기존 코드를 최소한으로 수정하여 에러를 해결한다.
전체 재작성이 아닌 **정밀 수정**에 집중한다.

## 절차

### 1. 실패 로그 수집

```bash
# PR 코멘트에서 QA 실패 보고 확인
gh pr view <PR번호> --comments --json comments --jq '.comments[-1].body'

# CI 로그 확인
gh run list --branch <브랜치> --limit 1 --json databaseId --jq '.[0].databaseId'
gh run view <run-id> --log-failed
```

### 2. 에러 분류

| 에러 유형 | 대응 전략 |
|-----------|-----------|
| 컴파일/빌드 에러 | 구문 오류, 타입 불일치 → 해당 라인 직접 수정 |
| 단위 테스트 실패 | 기대값/실제값 비교 → 로직 수정 또는 테스트 기대값 조정 |
| 통합 테스트 실패 | 모듈 간 인터페이스 불일치 → 설계 문서 재확인 |
| 린트/스타일 에러 | 자동 포매터 실행 또는 규칙에 맞게 수정 |
| 보안 스캔 실패 | 취약점 패턴 제거 (하드코딩 시크릿, 인젝션 등) |

### 3. 수정 원칙

- **최소 변경**: 실패한 부분만 수정한다. 주변 코드를 건드리지 않는다.
- **원인 추적**: 에러 메시지 → 실패 라인 → 원인 코드 순으로 추적한다.
- **회귀 방지**: 수정 후 관련 테스트를 로컬에서 먼저 실행한다.
- **커밋 분리**: 에러 수정 커밋은 기능 커밋과 분리한다: `fix(<scope>): <에러 설명>`

### 4. 수정 후 검증

```bash
# 로컬 테스트 실행 (run-tests 스킬 활용)
# 실패했던 특정 테스트만 먼저 실행
# 전체 테스트 통과 확인 후 푸시
git push
```

### 5. 결과 보고

PR에 수정 내용을 코멘트로 남긴다:

```markdown
## 에러 수정 보고

### 실패 원인
- [에러 메시지 요약]

### 수정 내용
- [파일:라인] 변경 설명

### 검증
- [x] 실패했던 테스트 통과
- [x] 전체 테스트 통과
```

## 규칙

- 3회 수정 시도 후에도 해결 못하면 `status:blocked` 라벨 + 수동 개입 요청한다.
- 테스트 자체가 잘못된 경우(flaky) QA에게 이슈로 보고한다.
- 설계 변경이 필요한 에러는 Architect에게 에스컬레이션한다.
